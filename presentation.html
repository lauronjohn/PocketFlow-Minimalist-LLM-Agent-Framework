1. Introduction — What is PocketFlow?

~100 lines of Python · Zero Dependencies · Zero Vendor Lock-in

PocketFlow is a minimalist LLM framework that models applications as nested directed graphs.

Four core primitives:

  • Node — unit of work (e.g., one LLM call)
  • Action — labeled edge deciding next node
  • Flow — orchestrates the graph walk
  • Shared Store — common data dictionary

Figure 1: PocketFlow vs. mainstream frameworks

PocketFlow is built on a bold thesis: all LLM frameworks are really doing the same thing — chaining LLM calls, passing data, handling failures. The authors captured this in ~100 Python lines using four primitives: Nodes do work, Actions route, Flows orchestrate, and a Shared Store carries state. Compared to LangChain's 405,000 lines, PocketFlow's core is literally a single init file.

Core Primitives — How They Fit Together

From these four building blocks, PocketFlow can express every common LLM design pattern — Agents, Multi-Agents, Workflows, RAG, Map-Reduce — without adding new framework code.

Everything specific to your app (LLM client, database, tools) is code you bring. The framework only orchestrates the graph.

Figure 2: Four core primitives and their relationships

The diagram on the right visualizes how Node, Action, Flow, and Shared Store relate. A Node produces an Action string; the Flow reads it and transitions. The Shared Store is the communication channel. The magic is that these four primitives compose to express everything from simple pipelines to complex multi-agent systems.

6. Context View

PocketFlow is an embedded library — not a service.

It draws a hard boundary:

  • Inside orchestration logic
  • Outside all integration logic

This "bring your own everything" boundary is why the install size is ~56 KB vs. competitors' 26–198 MB.

Figure 8: Context diagram

The context view shows PocketFlow's most important architectural boundary. The framework is a thin layer between the host application and the developer's utility functions. It never imports OpenAI or Pinecone — the developer writes a small call_llm() utility and invokes it inside a node. This constraint is why the dependency list is empty.

7. Logical View — Core Abstractions

Four abstractions, one base:

  • Node — unit of work
  • Action — routing result (a string)
  • Flow — graph orchestrator
  • Shared Store — communication state

Key insight: Node and Flow both inherit from BaseNode — this is what enables nested flows.

Figure 9: Logical view — class relationships

Figure 10: Node lifecycle — prep → exec → post

Node Lifecycle (3-step):

  1. prep(shared) — reads from Shared Store
  2. exec(prep_res) — computation; must not touch Shared Store
  3. post(shared, prep_res, exec_res) — writes & returns Action

Sealing exec() away from the Shared Store is what makes retries safe.

The logical view centers on four abstractions. The Node lifecycle is critical: prep reads, exec computes in isolation, post writes and routes. The exec step is deliberately sealed off from shared state — so if an LLM call fails, retrying doesn't corrupt anything. Node and Flow both inherit from BaseNode, which is the design move that makes nested flows possible.

7. Logical View — Actions, Flow & Nested Composition

Figure 11: Expense-approval flow with branching

Figure 12: Nested flows — sub-flows as reusable components

Action = a plain string returned by post(). Syntax: node_a - "approved" >> next_node

Flow walks the graph: runs current element → reads Action → follows transition → repeats.

Nested flows: A Flow itself is a BaseNode, so it can be dropped into another Flow. Build large pipelines from small, reusable sub-flows.

Actions drive routing. An expense-approval node returns "approved", "needs_revision", or "rejected" — and the Flow walks accordingly. Because transitions can point backwards, loops and agents emerge naturally. Nested flows are the compositional superpower: each box in figure 12 is an entire sub-flow, wired with the same >> syntax as individual nodes.

9. Process View — Runtime Execution & Failure Handling

Figure 14: Basic execution loop

Figure 15: Failure handling — retries & fallback

Execution: Single-threaded synchronous walk by default. Flow runs current element → reads Action → transitions → repeats.

Failure: Node retries exec() up to max_retries with wait intervals. On total failure, exec_fallback() provides a safe result. Because exec() is sealed from Shared Store, retries leave state consistent.

Advanced: Async, Batch, and Async Parallel Batch variants for I/O-bound workloads. Some language ports are sync-only — an active complexity frontier.

At runtime, PocketFlow runs a simple loop: execute node, read action, find next, repeat. Failure handling is localized to each node — retries happen inside exec(), which is isolated from shared state. This is why the framework can be so small: the process model is straightforward and doesn't need complex error-propagation machinery.

10. Deployment View

PocketFlow is a library, not a service — no daemon, no server, no prescribed topology.

Deployment shapes (from cookbook):

  • CLI tools — terminal-driven flows with human-in-the-loop
  • Web backends — FastAPI + WebSockets + SSE
  • Interactive apps — Streamlit with finite-state machines
  • Voice applications — VAD → STT → LLM → TTS pipeline

Framework footprint: ~56 KB, no special hardware requirements.

Figure 16: Deployment diagram

Deployment is entirely determined by the host application. PocketFlow imposes no runtime requirements beyond Python. You can embed it in a CLI script, a FastAPI server, or a Streamlit dashboard. External services like LLM APIs are reached over the network from inside node code. The deployment cost is dominated by those external services, not by PocketFlow itself.

Conclusion:

PocketFlow is best understood as a minimal, composable orchestration kernel.

Its real product is an abstraction — the bet that with the right four primitives, everything else is just composition.

Key trade-off: Complexity is not eliminated — it's relocated into user code and documentation. For small-to-medium apps and teams valuing transparency, this is an excellent bargain.

Technical debt is the mirror image of PocketFlow's strengths. The same minimalism that makes it elegant also offloads responsibility — for integrations, state hygiene, graph visibility, and observability — onto developers. In conclusion, PocketFlow demonstrates that a ~100-line, zero-dependency framework can express every common LLM pattern. The architecture is coherent because every decision serves the same goal: keep the core minimal and composable, push everything else out. Thank you. I'm happy to take questions.
